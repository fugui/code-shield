# 代码检视综合报告生成指令（深度代码检视）

你将收到一份 JSON 格式的代码检视发现清单（基于对整个仓库进行深度分析的发现）。请基于这些发现，生成一份指导开发团队进行缺陷修复与系统性重构的综合 Markdown 报告。

---

## 1. 输出语言与格式要求

*   **语言**：必须使用简体中文。
*   **格式**：Markdown 格式。
*   **结构**：报告必须严格由 **“一、检视结果概要”**、**“二、深度审查问题清单”**、**“三、系统性架构评估与建议”** 三个部分组成。

---

## 2. 章节详细要求

### 📌 一、检视结果概要
*   **内容**：
    *   检视目的与分析范围（代码仓名称、分支名称、扫描深度）。
    *   代码整体健康状况评估，以及发现的各类问题的汇总统计。
*   **【极其重要！硬性约束条件，不得违反】**：
    *   本章节的最后必须输出一行为机器解析设计的统计数据，格式必须是：
        `致命：N，严重：N，一般：N，建议：N`
    *   **限制条件**：
        1.  关键字（致命、严重、一般、建议）不得修改，顺序不得颠倒。
        2.  冒号必须是中文冒号 `：`，多个项之间的分隔符必须是中文逗号 `，`。
        3.  N 代表相应的非负整数数值，如果没有此类问题填 `0`。
        4.  不要包含任何多余的前导空格或后缀字符。

### 📌 二、深度审查问题清单
*   **排序要求**：必须按照危害严重程度从高到低进行排列，即：**致命 ➔ 严重 ➔ 一般 ➔ 建议**。
*   **格式结构**：每个检出意见须统一编号，并采用以下标准化布局：
    #### [编号]. [简短清晰的问题标题]
    *   **严重级别**：`致命` | `严重` | `一般` | `建议`
    *   **问题分类**：输出分析阶段指定的 `主分类-二级分类`
    *   **代码位置**：`[文件相对路径]:[精准行号]`
    *   **现象描述**：深入解析为什么这行代码是错误的/存在隐患的，在什么特定环境或数据流下可能被触发。
    *   **风险影响**：该缺陷对程序运行时的具体影响（如崩溃、资源耗尽、死锁、性能严重抖动等）。
    *   **重构建议**：给出精准的修复策略，强烈建议**使用 Markdown 代码块提供直观的修复代码示例（C++/Go/Python/Java）**，以便开发者开箱即用。
    *   **设计参考**（可选）：相关的业内标准规范（例如 *C++ Core Guidelines*，*CWE-XYZ* 编号，*Effective Java* 等）。

### 📌 三、系统性架构评估与建议
*   **架构总结**：针对全量扫描暴露出的架构问题进行高层次提炼（如：“系统存在严重的循环依赖”、“全局静态变量过多导致重入风险”）。
*   **长效治理措施**：给团队的技术债务偿还建议、设计模式的引入、或后续需要专项跟进的技术文档/设计会签。

## 3. 报告示例模板

```markdown
# 深度代码检视报告

## 一、检视结果概要

本次代码审查为针对常驻后台服务的全量存量代码深度体检，着重评估系统在高并发与长期运行条件下的内存安全、多线程并发安全以及模块间耦合度。

*   **检视仓库**：ilxdrv/core-engine
*   **检视分支**：master
*   **评估深度**：全量代码及跨模块关联分析

### 检视结果概要
致命：1，严重：2，一般：5，建议：1

---

## 二、深度审查问题清单

### 致命

#### 1. 多线程异常退出路径下的空指针解引用
*   **严重级别**：致命
*   **问题分类**：逻辑与正确性-并发安全
*   **代码位置**：`src/module_a/data_processor.cpp:87`
*   **现象描述**：函数 `processData()` 在独立线程中处理输入流，异常退出路径直接对传入的共享上下文指针进行析构清理，但在调用前未进行有效性检查。如果上下游生命周期不一致，将发生致命访问违规。
*   **风险影响**：直接导致后台引擎核心进程崩溃挂掉。
*   **重构建议**：
  ```cpp
  if (data_ptr == nullptr) {
      LOG_ERROR("data_ptr is null. Aborting task processing.");
      return ERROR_INVALID_INPUT;
  }
  ```

### 严重

#### 2. ScopeGuard缺失导致异常退出路径句柄泄露
*   **严重级别**：严重
*   **问题分类**：性能与运行效率-资源未释放
*   **代码位置**：`src/module_b/file_handler.cpp:156`
*   **现象描述**：`file_fd` 是在函数顶部申请的底层系统描述符。在解析配置数据失败的分支中，函数直接执行了 `return false`，遗漏了文件句柄的显式关闭。
*   **风险影响**：在持续不断的配置更新请求下，将迅速耗尽文件描述符配额，导致后续所有 I/O、Socket 链接创建失败。
*   **重构建议**：使用 C++11 RAII 思想进行包装防御：
  ```cpp
  class FileGuard {
  public:
      explicit FileGuard(int fd) : fd_(fd) {}
      ~FileGuard() { if (fd_ >= 0) close(fd_); }
  private:
      int fd_;
  };
  FileGuard guard(file_fd);
  ```

## 三、系统性架构评估与建议

### 1. 架构级坏味道总结
全量扫描显示，当前系统的主要技术债集中在**模块生命周期管理不清晰**。多处模块在跨线程通信时直接传递裸指针，且缺乏清晰的析构边界定义，这极易引发野指针和多线程竞态。

### 2. 改进路线建议
1.  **引入 Scope 级别生命周期控制**：建议对全部底层 C++ 模块进行智能指针（`std::shared_ptr`/`std::unique_ptr`）升级，彻底消灭手动 `delete`。
2.  **治理循环依赖**：`module_a` 与 `module_b` 存在强双向调用，建议抽象出 `IModuleCallback` 虚基类，使用接口解耦。
