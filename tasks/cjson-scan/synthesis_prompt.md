# cJSON 内存泄漏代码检视综合报告生成指令

你将收到一份 JSON 格式的出JSON内存泄漏代码检视发现清单（基于 @analysis_prompt.md 提示词进行的检视）。请基于这些发现，生成一份指导开发者进行修复的综合 Markdown 报告。

## 输入说明

输入是一个 JSON 数组，每个元素包含以下字段：
- `severity`: 问题严重程度（致命/严重/一般/建议）
- `category`: 问题分类
- `file_path`: 文件路径
- `line_number`: 行号
- `code_snippet`: 原始代码片段
- `title`: 问题标题
- `detail`: 详细描述
- `suggestion`: 修复建议

## 输出格式要求

请使用简体中文，以 Markdown 格式输出，内容包括"概要"、"检视问题清单"、"总结"三部分组成。

### 一、概要
- 检视目的和方法的简要介绍

> ⚠️ **概要部分的最后一行，格式约束（机器解析，不得违反）**：
> - 必须严格使用 `致命：N，严重：N，一般：N，建议：N` 的格式，关键字和标点均不得更改
> - 冒号使用中文冒号 `：`，分隔符使用中文逗号 `，`
> - N 为非负整数，无问题填 `0`

### 二、检视问题清单
- 请按照 **致命 -> 严重 -> 一般 -> 建议** 的顺序排列
- 每个检视意见须有编号
- 每个问题包含以下字段：
  - **级别**：问题定级
  - **位置**：文件路径和行号
  - **描述**：问题描述
  - **影响**：问题可能造成的影响
  - **建议**：修改建议，最好是示例代码片段或伪代码级别的修改思路
  - **参考**（可选）：相关的最佳实践或文档链接
- 问题定位精度到行级

### 三、总结
- 对检视的问题进行总结、修复建议
- 对代码整体修改质量进行一个简短的总结
- 如果问题比较严重，可以给出人员和组织需要进行的学习和培训的建议
- 需要跟进的事项（如需要补充的文档、需要进一步讨论的设计问题）
