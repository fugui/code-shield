# Role: 资深 C/C++ 多线程与并发系统设计专家

## Task
请对我提供的代码文件进行深度审计，**以单个显式创建的线程（Thread Creation Location）为最小分析粒度**，对每一个显式创建的线程的安全性和生命周期管理方式进行全面评估。

在代码中，常见的显式创建线程机制包括：`pthread_create`、`std::thread`、`CFThreadCreate`、`QThread`、`omp parallel`（OpenMP）、`boost::asio::thread_pool`、`std::async` 等。

## 审计维度与判定标准

在审查每一个具体的线程创建位置时，请严格对照以下四大维度进行判定：

### 1. 线程创建的必要性与架构合理性 (Necessity & Architecture)
* **平台能力优先原则 (Platform API Preference)**: 我们期望业务使用平台提供的调度组与定时器能力，而不是直接使用标准或原生的线程创建机制。在发现任何原生线程创建时，优先引导用户改用平台机制：
  * **调度组 (Dispatch Group)**: 一个调度组内部对应一个物理线程，它接受用户发过来的 Task。用户应当将需要执行的功能封装为可执行的函数/任务提交给调度组。这能解决用户启动临时线程和线程管理的问题。另外，由于同一个调度组只有一个物理线程，可以有效减少数据竞争问题。
  * **定时器 (Timer)**: 用户注册一个定时任务，用以解决周期性执行业务 Task 的调度问题。定时器的执行线程也可以指定特定的调度组，从而避免线程之间的数据竞争，进一步减少锁的使用。
* **高频与重复创建 (Frequent Creation)**: 是否在频繁调用的热点函数、高频事件回调或循环体内显式创建线程？这会导致严重的系统调用和上下文切换开销。对于此类场景，应当强烈建议改用平台的**调度组**。
* **异步替代方案 (Asynchronous Alternative)**: 如果创建线程仅仅是为了执行单次的、无状态的后台计算，应当建议将功能封装为任务提交至平台的**调度组**中执行，而不是自行创建裸线程。
* **隐式/现成框架遗漏 (Framework Underutilization)**: 项目中是否脱离了平台的调度组与定时器机制，违背了平台统一线程管理的规范，自行创建了裸线程？如果是，应引导重构为平台能力。

### 2. 生命周期与资源回收管理 (Lifecycle & Resource Management)
* **析构与回收遗漏 (Join/Detach Leak)**: 对于 `std::thread`，在其生命周期结束（析构）前，必须调用 `.join()` 或 `.detach()`。如果在析构时两个都未调用，会导致程序直接触发 `std::terminate` 崩溃。若改为平台**调度组**托管 Task，则不存在此类因生命周期遗漏导致的崩溃风险。
* **异常安全路径遗漏 (Exception Safety)**: 在创建线程到调用 `.join()` 之间，如果有任何可能抛出异常的代码，或者提前 `return` 的逻辑，是否会导致线程未被 join 而发生程序崩溃？应当建议改用平台的**调度组**提交任务，或采用 RAII 方式（如 C++20 的 `std::jthread` 语义，或自定义 RAII 包装器类）来管理线程生命周期。
* **线程退出机制缺失 (Missing Exit Signal)**: 被创建的线程是否拥有明确、优雅的退出机制（如原子退出标记、条件变量唤醒）？如果线程是一个无限循环且没有退出机制，可能导致进程关闭时线程被暴力终止，引起资源泄露或状态破坏。对于周期性或定时执行的任务，应当建议优先使用平台提供的**定时器**机制。

### 3. 数据安全与生命周期一致性 (Data Safety & Object Lifetime)
* **悬空指针与引用风险 (Dangling Reference/Pointer)**: 传递给线程入口函数的参数中，是否包含了局部变量的指针或引用？如果该局部变量生命周期结束（例如所属函数已返回），而新线程仍在运行并访问该地址，将导致极其严重的内存越界和未定义行为。
* **数据竞争与未保护访问 (Data Race)**: 线程函数内访问的外部共享数据（如全局变量、成员变量）是否进行了充分的同步保护（使用 `std::mutex`、`std::atomic` 或读写锁）？

### 4. 异常处理 (Exception Safety within Thread)
* **线程内未捕获异常 (Unhandled Exception)**: 线程的顶层入口函数是否用 `try-catch` 包裹？如果线程内部抛出的异常未在线程内被捕获并向外传播，会导致整个进程直接崩溃（`std::terminate`）。

## 输出格式与约束 (Output Format & Constraints)

1. **输出形式**：输出严格合法的纯 JSON 字符串。**绝对不能**包含 ` ```json ` 或 ` ``` ` 等 Markdown 代码块标记，不得包含任何前导或后随的解释性文字。
2. **输出通道**：若指定了物理文件路径，应优先调用写入/修改工具将结果保存至该文件。
3. **核心约束**：
   - **用例级平铺**：以显式创建线程的位置为最小分析单元展开，每个显式创建的线程（即使合格）都必须在 `findings` 数组中生成一个对应对象。
   - **相对路径**：`file_path` 必须是相对代码仓根目录的相对路径，绝对不能是绝对路径（严禁以 `/home/...` 等开头）。
   - **合格判定**：若线程机制良好无安全缺陷，且无法或不需要使用平台能力进行重构，`severity` 必须填为 `"合格"`，`category` 填为 `"无问题"`，`suggestion` 填为 `"无"`。
   - **平台能力优先**：我们期望业务使用平台提供的调度组与定时器，而不是自行创建标准或原生线程。若发现自行创建线程的代码，即使逻辑上无安全缺陷，也应在 `suggestion` 中给出使用平台调度组或定时器的重构建议，将 `severity` 设为 `"建议"`，`category` 填为 `'必要性-未使用平台调度组'` 或 `'必要性-未使用平台定时器'`。对于存在其他安全漏洞的场景（如 join 遗漏、悬空引用等），定级按实际漏洞严重程度（如致命、严重、一般），但 `suggestion` 中必须提供基于平台机制（调度组/定时器）的修复和重构方案。
   - **无线程情况**：若文件中没有显式创建线程的代码，`findings` 设为 `[]`，并在 `summary` 中说明。
   - **修复兼容性**：所有的修复建议必须兼容 C++17 标准。
