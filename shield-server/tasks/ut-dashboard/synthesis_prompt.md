# UT代码质量评估 报告总结指令

你将收到一份 JSON 格式的代码检视发现清单。请基于这些发现（基于 @analysis_prompt.md 提示词进行的分析），生成一份面向技术管理者的综合 Markdown 报告。

## 输入说明

输入是一个 JSON 数组，每个元素包含以下字段：
{
  "findings": [
    {
      "file_path": "测试用例所在的相对文件路径",
      "line_number": "[10-23]",
      "title": "测试函数的完整名称",
      "detail": "精炼描述该用例的测试目的（它在验证什么输入、什么边界或什么行为）",
      "severity": "严重程度， 合格|阻塞|严重|主要|提示|建议",
      "category": "无问题|断言有效性-空测试|断言有效性-永真断言|断言有效性-无效捕获|断言有效性-Mock遗漏|提前返回-非法退出|提前返回-静默跳过|功能单一性-面条测试|功能单一性-路径混合|其它",
      "code_snippet": "相关代码片段",
      "suggestion": "修复建议"
    }
  ]
}

## 输出格式

请使用简体中文，以 Markdown 格式输出。报告需包含以下三部分：

### 一、 审计概要
* 介绍审计的目标和策略， 并介绍统计扫描的代码仓的情况（代码仓名称、分支、最新的Commit ID 等）;
* 在本部分的最后部分， 严格采用如下格式（机器解析，不得违反）进行总结：

检视问题情况:  用例总数：15个，合格：5个，阻塞：1个，严重：0个，主要：1个，提示：0个，建议：8个

### 二、 问题清单 (按发现顺序逐条列出)
**[问题 #编号] - [文件名]**
* **测试方法**：`MethodName`
* **缺陷类型**：(断言有效性 / 提前返回 / 功能单一性)
* **严重程度**：（完美|阻塞|严重|主要|提示|建议）
* **现状描述**：详细描述该测试代码存在的具体缺陷， 尽可能把问题所在的源代码片段添加上。
* **改进建议**：给出具体的重构或修复方案， 尽可能给出可直接实施的源代码。

--- *(每条问题之间使用分割线， 类似的问题不得压缩输出)* ---

### 三、 总结
* 对该代码仓的整体测试质量进行评价。
* 提供针对性的改进建议（如引入特定的静态检查工具、规范化 TDD 流程等）。
