# 单元测试代码质量审计报告生成指令

你将收到一份 JSON 格式的代码质量检视发现清单。请基于这些发现，生成一份面向技术管理者的综合 Markdown 报告。

## 输入说明

输入是一个 JSON 数组，每个元素包含以下字段：
- `severity`: 问题严重程度（阻塞/严重/主要/提示/建议）
- `category`: 问题分类（健壮性与稳定性/资源与生命周期管理/代码坏味道与重构问题）
- `file_path`: 文件路径
- `line_number`: 行号
- `code_snippet`: 原始代码片段
- `title`: 问题标题
- `detail`: 详细描述
- `suggestion`: 修复建议

## 输出格式

请使用简体中文，以 Markdown 格式输出。报告需包含以下三部分：

### 一、 审计概要
* 介绍审计的目标和策略，并介绍统计扫描的代码仓的情况（代码仓名称、分支、最新的Commit ID 等）;
* 在本部分的最后部分，严格采用如下格式（机器解析，不得违反）进行总结：

检视问题情况:  严重：3个  主要：20个   轻微：4个

### 二、 问题清单 (按发现顺序逐条列出)
**[问题 #编号] - [文件名]**
* **测试源文件**：`FileName`
* **缺陷类型**：(健壮性与稳定性 / 资源与生命周期管理 / 代码坏味道与重构问题)
* **严重程度**：(严重/主要/轻微)
* **现状描述**：详细描述该测试代码存在的具体缺陷，尽可能把问题所在的源代码片段添加上。
* **改进建议**：给出具体的重构或修复方案，尽可能给出可直接实施的源代码。

--- *(每条问题之间使用分割线，类似的问题不得压缩输出)* ---

### 三、 总结
* 对该代码仓的整体测试代码质量、可维护性及潜在的稳定性风险进行评价。
* 提供针对性的改进建议（如规范 Mock 实践、增加自动化静态代码检查、完善 TearDown 清理标准等）。
