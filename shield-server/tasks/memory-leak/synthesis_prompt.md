# 代码检视综合报告生成指令

你将收到一份 JSON 格式的代码检视发现清单（基于 @analysis_prompt.md 提示词进行的检视）。请基于这些发现，生成一份指导开发者进行修复的综合 Markdown 报告。

## 输入说明

输入是一个 JSON 数组，每个元素包含以下字段：
- `severity`: 问题严重程度（阻塞/严重/主要/提示/建议）
- `category`: 问题分类
- `file_path`: 文件路径
- `line_number`: 行号
- `code_snippet`: 原始代码片段
- `title`: 问题标题
- `detail`: 详细描述
- `suggestion`: 修复建议

## 输出格式要求

请使用简体中文，以 Markdown 格式输出，内容包括"概要"、"检视问题清单"、"总结"三部分组成。

### 一、概要
- 检视目的和方法的简要介绍
- 检视的范围：代码仓、分支、commit hash 的范围、修改日期的区间， 主要代码作者
- 代码量：新增、删除了多少代码
- 主要新增、修改、去除的功能/特性总结
- **特别说明**：如果修改了对外接口，要明确指出来
- 阻塞：0，严重：0，主要：0，提示：0，建议：0

> ⚠️ **概要部分的最后一行，格式约束（机器解析，不得违反）**：
> - 必须严格使用 `阻塞：N，严重：N，主要：N，提示：N，建议：N` 的格式，关键字和标点均不得更改
> - 冒号使用中文冒号 `：`，分隔符使用中文逗号 `，`
> - N 为非负整数，无问题填 `0`

### 二、检视问题清单
- 请按照 **阻塞 -> 严重 -> 主要 -> 提示 -> 建议** 的顺序排列
- 每个检视意见须有编号
- 每个问题包含以下字段：
  - **级别**：问题定级
  - **位置**：文件路径和行号
  - **描述**：问题描述
  - **影响**：问题可能造成的影响
  - **建议**：修改建议，最好是示例代码片段或伪代码级别的修改思路
  - **参考**（可选）：相关的最佳实践或文档链接
- 问题定位精度到行级

### 三、总结
- 对检视的问题进行总结、修复建议
- 对代码整体修改质量进行一个简短的总结
- 如果问题比较严重，可以给出人员和组织需要进行的学习和培训的建议
- 需要跟进的事项（如需要补充的文档、需要进一步讨论的设计问题）

## 审查报告示例模板

```markdown
# 代码审查报告

## 一、概要

本次审查涉及 commit `abc1234` 到 `def5678`，修改日期区间为 2026-03-01 至 2026-03-15。

### 代码变更统计
- 新增代码：1,234 行
- 删除代码：567 行
- 修改文件：12 个

### 主要变更内容
- 新增：模块 A 的实现，包含数据采集和处理功能
- 修改：模块 B 的接口，增加了参数校验
- 删除：废弃的模块 C

### 特别说明
- 修改了对外接口 `ModuleB::process()`，参数列表发生变化，需要通知调用方

### Commit 信息
| Commit ID | 作者 | 提交日期 | 主要修改内容 |
|----------|------|----------|--------------|
| 8b12038  | my@company.com | 2026-03-05 | 模块 A 的实现，包含数据采集和处理功能 |

### 检出问题数量
阻塞：1，严重：2，主要：3，提示：2，建议：0

## 二、检视问题清单

### 阻塞

#### 1. 空指针解引用风险
- **级别**：阻塞
- **位置**：`src/module_a/data_processor.cpp:87`
- **描述**：函数 `processData()` 在使用指针 `data_ptr` 之前没有进行空指针检查
- **影响**：可能导致程序崩溃
- **建议**：
  ```cpp
  if (data_ptr == nullptr) {
      LOG_ERROR("data_ptr is null");
      return ERROR_INVALID_INPUT;
  }
  ```
- **参考**：C++ Core Guidelines C.60

### 严重

#### 2. 资源泄漏
- **级别**：严重
- **位置**：`src/module_b/file_handler.cpp:156`
- **描述**：文件句柄 `file_fd` 在异常路径下可能未被关闭
- **影响**：可能导致文件描述符耗尽
- **建议**：使用 RAII 模式管理资源
  ```cpp
  class FileGuard {
  public:
      explicit FileGuard(int fd) : fd_(fd) {}
      ~FileGuard() { if (fd_ >= 0) close(fd_); }
  private:
      int fd_;
  };
  FileGuard guard(file_fd);
  ```

### 主要

#### 3. 函数复杂度过高
- **级别**：主要
- **位置**：`src/module_a/data_processor.cpp:45-234`
- **描述**：函数 `processData()` 的圈复杂度为 28，认知复杂度为 25，建议拆分
- **影响**：降低代码可读性和可维护性
- **建议**：将函数拆分为多个更小的函数，每个函数负责单一职责

## 三、总结

本次审查共发现问题 8 个，其中 阻塞 1 个，严重 2 个，主要 3 个，提示 2 个。

建议优先修复 阻塞 和 严重 级别的问题。代码整体质量良好，但需要加强空指针检查和资源管理。

组织层面建议：
1. 开展 RAII 编程模式的培训
2. 建立代码复杂度检查的 CI 规则
```
