# 代码检视综合报告生成指令（近期变更检视）

你将收到一份 JSON 格式的代码检视发现清单（基于对代码仓库最近提交变更的分析）。请基于这些发现与输入的修改信息，生成一份专业的增量变更代码审查 Markdown 报告，指导开发团队进行缺陷修复、排除合入风险。

---

## 1. 输出语言与格式要求

*   **语言**：必须使用简体中文。
*   **格式**：Markdown 格式。
*   **结构**：报告必须严格由 **“一、变更与检视结果概要”**、**“二、变更缺陷与风险清单”**、**“三、合并与改进建议”** 三个部分组成。

---

## 2. 章节详细要求

### 📌 一、变更与检视结果概要
*   **内容要求**：
    1.  **基本信息**：审查的 Commit 区间、修改日期的区间、主要作者。
    2.  **代码变更统计**：新增代码行数、删除代码行数、被修改文件数。
    3.  **主要变更内容**：总结本次提交或合并中，主要新增、修改、去除了哪些功能/特性。
    4.  **【特别说明】对外接口变更**：如果本次提交修改了对外公共接口（API、SDK函数签名、暴露的 RPC 方法），**必须以醒目的高亮形式（如 Blockquote 或 Warning Alert）明确指出来**。
*   **【极其重要！硬性约束条件，不得违反】**：
    *   本章节的最后必须输出一行为机器解析设计的统计数据，格式必须是：
        `阻塞：N，严重：N，主要：N，提示：N，建议：N`
    *   **限制条件**：
        1.  关键字（阻塞、严重、主要、提示、建议）不得修改，顺序不得颠倒。
        2.  冒号必须是中文冒号 `：`，多个项之间的分隔符必须是中文逗号 `，`。
        3.  N 代表相应的非负整数数值，如果没有此类问题填 `0`。
        4.  不要包含任何多余的前导空格或后缀字符。

### 📌 二、变更缺陷与风险清单
*   **排序要求**：必须按照危害严重程度从高到低进行排列，即：**阻塞 ➔ 严重 ➔ 主要 ➔ 提示 ➔ 建议**。
*   **格式结构**：每个检出意见须统一编号，并采用以下标准化布局：
    #### [编号]. [简短清晰的问题标题]
    *   **严重级别**：`阻塞` | `严重` | `主要` | `提示` | `建议`
    *   **问题分类**：输出分析阶段指定的 `主分类-二级分类`
    *   **代码位置**：`[文件相对路径]:[精准行号]`
    *   **现象描述**：深入解析本次增量提交中引入的代码逻辑为什么有缺陷/风险。
    *   **回归危害**：该缺陷如果被合入，对现有系统的已有功能有什么影响？是否可能引入并发数据竞争、引发运行异常或系统崩溃？
    *   **重构建议**：给出精准的修复策略，强烈建议**使用 Markdown 代码块提供直观的修复代码示例（C++/Go/Python/Java）**，以便开发者开箱即用。

### 📌 三、合并与改进建议
*   **合入结论**：明确评估本次提交是否建议安全合入，还是应当由于存在“阻塞/严重”缺陷而直接进行阻断阻退（Reject）。
*   **持续改进**：针对本次变更显露出的代码质量薄弱点，提供短平快的具体改进建议（如：“增强单测覆盖率”、“补充外部接口文档说明”）。

# 近期变更代码审查报告

## 一、变更与检视结果概要

本次审查涉及 commit `abc1234` 到 `def5678`，修改日期区间为 2026-05-20 至 2026-05-27。

### 📊 代码变更统计
*   **新增代码**：1,234 行
*   **删除代码**：567 行
*   **修改文件**：12 个

### 🛠️ 主要变更内容
*   **新增**：模块 A 的异步采集流实现。
*   **修改**：重构了 `module_b` 的数据刷新频率限制逻辑。

> ⚠️ **【重要特别说明】公共接口变更**
> 修改了公共 API `ModuleB::SetSyncInterval(int milliseconds)`，将参数单位由秒改为毫秒，该变更为**不向下兼容的破坏性修改**，已依赖此接口的下游模块需同步修改参数值，否则会引发严重限流！

### 检视结果概要
阻塞：1，严重：1，主要：2，提示：0，建议：0

## 二、变更缺陷与风险清单

### 阻塞

#### 1. 新增 API 暴露未捕获的 NullPointerException 隐患
*   **严重级别**：阻塞
*   **问题分类**：接口兼容性-上下游参数失配
*   **代码位置**：`src/module_a/data_processor.java:45`
*   **现象描述**：本次提交在新增的 `registerListener` 对外方法中，直接将传入的 `listener` 放入全局注册表中并执行了回调方法 `listener.onInit()`，但完全缺失了对 `listener` 是否为空的校验防御。
*   **回归危害**：一旦下游在调用此接口时传空，将立即抛出 `NullPointerException` 异常，可能导致整个监听服务子系统崩溃退出。
*   **重构建议**：
  ```java
  public void registerListener(Listener listener) {
      if (listener == null) {
          throw new IllegalArgumentException("listener cannot be null");
      }
      this.listeners.add(listener);
      listener.onInit();
  }
  ```

---

## 三、合并与改进建议

### 1. 合入结论
**【建议暂缓合入】**：本次变更由于存在公共 API 破坏性向下兼容改动，且暴露了阻塞级空指针风险，**强烈建议暂缓合并**。请开发人员根据本报告修复上述 `阻塞` 及 `严重` 级缺陷，并同步通知下游模块调整参数单位后，再行重新发起合并审查。

### 2. 改进建议
1.  **补充 API 升级文档**：公共接口参数单位变更为毫秒，需显式在接口设计文档及 Changelog 中高亮声明。
2.  **强化 Null-Safety 防御**：开发团队应严格执行强防御性编程规范，任何对外部公共入参的使用前必须实施非空或有效性边界校验。
```
